# 포트폴리오 프로젝트 — 작업 규칙

> 이 파일은 모든 AI 어시스턴트가 이 포트폴리오 프로젝트에서 작업할 때 따라야 할 전체 규칙이다.

---

## 기술 스택

- **프레임워크**: Next.js (App Router)
- **언어**: JavaScript (TypeScript 사용 금지)
- **스타일링**: SCSS Modules (`.module.scss`)
- **의존성**: `sass` 패키지 필요

---

## 파일 구조

### 프로젝트별 콜로케이션 (필수)

각 프로젝트의 모든 관련 코드는 `src/app/projects/{name}/` 안에 둔다.

```
src/app/projects/{name}/
├── _components/    # 프로젝트 전용 컴포넌트
├── _styles/        # 프로젝트 전용 스타일/토큰
├── _lib/           # 프로젝트 전용 유틸리티
├── _hooks/         # 프로젝트 전용 훅
├── layout.js
├── page.js
└── {routes}/
```

- `_` 접두사 폴더는 Next.js 라우팅에서 제외됨
- 프로젝트의 **소스 코드**(컴포넌트, 스타일, 유틸, 훅)를 `src/app/projects/{name}/` 바깥에 두지 않는다. 문서/데이터는 별개.

### 포트폴리오 공용 컴포넌트

여러 프로젝트에서 공유하는 컴포넌트만 `src/components/portfolio/`에 둔다.
코딩 컨벤션(PascalCase 폴더, SCSS Modules, kebab-case 클래스명)은 공용 컴포넌트에도 동일하게 적용한다.

```
src/components/portfolio/
├── Header/
├── Footer/
```

---

## 코딩 컨벤션

- **컴포넌트**: PascalCase 폴더 + 동일 이름 .js / .module.scss
  - 예: `AllergyBanner/AllergyBanner.js` + `AllergyBanner/AllergyBanner.module.scss`
- **파일명**: 컴포넌트 = PascalCase, 유틸/훅 = camelCase
- **SCSS 클래스명**: kebab-case, Apple 스타일 (`.unit-wrapper`, `.unit-copy-wrapper`)
  - JS에서 접근: `styles['unit-wrapper']` (bracket notation)
- **SCSS 구조**: 컴포넌트별 `.module.scss`는 해당 컴포넌트 폴더 안에 함께 배치. 디자인 토큰/전역 스타일만 `_styles/`에 둔다.
- **JS 변수/함수명**: camelCase
- **언어**: 변수/함수명 영어, 주석 한국어, UI 텍스트는 한국어 기본 (상황에 따라 사용자가 결정)
- **Server/Client**: 기본은 Server Component, 인터랙션 필요한 부분만 `'use client'`

## 코드 원칙

- **간결성**: 코드는 간결하고 가독성 좋게 작성한다. 불필요한 추상화를 피한다.
- **수정 용이성**: 사용자가 직접 수정할 수 있도록 복잡한 패턴을 지양한다.
- **주석**: 코드만으로 의도가 불분명한 부분에 상세한 주석을 단다. 자명한 코드에는 불필요한 주석을 달지 않는다.

---

## AI 작업 원칙

- **비판적 사고**: 사용자의 요청에 대해 논리적이고 비판적으로 사고하고 판단한다. 무조건 동의하지 않는다.
- **이해 우선**: 기존 코드를 먼저 읽고 이해한 후 수정한다. 맥락 없이 코드를 생성하지 않는다.
- **최소 변경**: 변경 범위를 최소화한다. 요청받지 않은 부분을 임의로 수정하지 않는다.
- **확인 후 실행**: 되돌리기 어려운 작업(파일 삭제, 구조 변경 등)은 사용자에게 먼저 확인한다.

## 모델 분업 원칙

- **계획(Planning)**: Opus로 진행 (`/plan` 모드)
- **코드 작성(Implementation)**: Sonnet으로 진행
- **계획 완료 후**: 반드시 사용자에게 "Sonnet으로 전환 후 실행해 주세요" 라고 안내한다.
- **Opus는 코드를 작성하지 않는다.** 계획만 수립하고, 구현은 반드시 Sonnet에게 넘긴다.

### UI 우선 구현 (필수)

모든 구현은 **UI(마크업+스타일) 먼저, 기능(인터랙션+로직) 나중에** 순서를 따른다.

1. 먼저 정적 UI를 완성한다 (HTML 구조, SCSS 스타일, 레이아웃)
2. UI가 확정된 후 기능을 추가한다 (상태 관리, 이벤트 핸들러, 데이터 연동 등)

### 단계적 구현 (필수)

Sonnet은 한 번에 전체를 구현하지 않는다. 아래 규칙을 따른다.

1. **한 단계 = 페이지 1개 또는 기능 1개** (예: 패널 프레임, 대시보드 뷰, 차트 뷰 등)
2. **단계마다 빌드 확인** — `next build` 성공을 확인한 후 다음 단계로 진행
3. **단계마다 사용자 확인** — 결과를 보고하고, 사용자가 승인하면 다음 단계 진행
4. **계획의 구현 순서를 따른다** — Opus가 정한 순서를 임의로 건너뛰지 않는다

## Devil's Advocate 검토 (필수)

> **모든 작업의 마지막 단계에서 반드시 Devil's Advocate 검토를 수행한다.**

### 검토 항목

1. **코드 품질**: 미사용 변수, 불필요한 import, 중복 코드, 하드코딩된 값
2. **구조 준수**: 콜로케이션 규칙 위반 여부 (프로젝트 코드가 해당 프로젝트 폴더 바깥에 없는지)
3. **접근성**: WCAG 2.2 AA, 시맨틱 HTML, 키보드 내비게이션, 색상만으로 정보 전달 금지
4. **반응형**: 레이아웃 깨짐 없는지, 프로젝트별 UI 제약 준수 여부
5. **데이터 정합성**: JSON 데이터 ↔ 컴포넌트 props 매핑이 정확한지
6. **이음 콘텐츠 작업 시**: `src/app/projects/eum/CLAUDE.md`의 Devil's Advocate 항목도 추가 수행 (용어, 법적 준수, 플로우 일관성 등)

### 검토 방식

- 발견된 문제를 **표 형태로 정리** (파일 | 문제 | 수정 방향)
- 정리 후 **즉시 수정 반영**
- 수정 완료 후 **수정 요약 제시**
